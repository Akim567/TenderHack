# План построения backend-каркаса для хакатона

## 1. Базовая архитектура проекта
- Выбрать стиль: модульный монолит (оптимально для хакатона).
- Разделить слои: `api`, `services`, `repositories`, `schemas`, `core`, `models`.
- Зафиксировать правила импортов: верхние слои не зависят от нижних циклически.

## 2. Структура директорий (скелет)
- Создать основу:
  - `app/main.py` (точка входа FastAPI)
  - `app/api/v1/` (роуты)
  - `app/core/` (конфиг, логгер, зависимости)
  - `app/schemas/` (Pydantic-схемы запросов/ответов)
  - `app/services/` (бизнес-слой, пока с заглушками)
  - `app/repositories/` (доступ к данным, пока интерфейсы/заглушки)
  - `app/models/` (ORM-модели/доменные модели)
  - `tests/` (unit + integration каркас)
- Добавить `__init__.py` в пакеты.

## 3. Конфигурация окружения
- Ввести `Settings` через `pydantic-settings`.
- Подготовить `.env.example` с переменными:
  - `APP_NAME`, `APP_ENV`, `DEBUG`, `HOST`, `PORT`
  - `DATABASE_URL`, `REDIS_URL` (если нужны)
  - `LOG_LEVEL`
- Вынести чтение настроек в `app/core/config.py`.

## 4. Инициализация приложения
- В `app/main.py` оформить factory-подход: `create_app()`.
- Подключить:
  - healthcheck-ручку (`/health`)
  - versioned router (`/api/v1`)
  - кастомные exception handlers
  - middleware (request-id, CORS, базовое логирование)
- Оставить бизнес-ручки пустыми, но с корректными контрактами.

## 5. Контракты API (без логики)
- Для каждой будущей фичи создать:
  - файл роутов в `app/api/v1/`
  - входные/выходные схемы в `app/schemas/`
  - заглушку service-метода в `app/services/`
- Возвращать `501 Not Implemented` или фиктивный ответ с `TODO`.
- Сразу проставить теги, summary и response-модели для OpenAPI.

## 6. Слой зависимостей и DI
- Создать `app/core/dependencies.py`.
- Описать провайдеры зависимостей:
  - settings
  - db session (заглушка)
  - сервисы (через `Depends`)
- Избежать прямого создания сервисов внутри роутов.

## 7. Подготовка инфраструктуры данных (каркас)
- Подключить SQLAlchemy/SQLModel (если планируется БД) только на уровне каркаса.
- Создать файл `app/core/db.py` с engine/sessionmaker.
- Подготовить папку миграций (`alembic/`) и базовый `alembic.ini`.
- Не реализовывать реальную доменную схему до согласования требований.

## 8. Логирование и наблюдаемость
- Настроить единый логгер (`app/core/logging.py`).
- Единый формат логов для локалки и прод-режима.
- Добавить middleware для логирования запроса/ответа (минимально).

## 9. Тестовый каркас
- Настроить `pytest` + `httpx`/`TestClient`.
- Добавить:
  - smoke-тест на запуск приложения
  - тест на `/health`
  - шаблон теста для будущих роутов
- Подготовить фикстуры `tests/conftest.py`.

## 10. Качество кода и стандарты
- Подключить линтеры/форматтеры:
  - `ruff` (lint + format)
  - `mypy` (по желанию, хотя бы базово)
- Добавить `pre-commit` с минимальными хуками.

## 11. Docker и запуск
- Подготовить:
  - `Dockerfile` (backend service)
  - `docker-compose.yml` (app + db/redis при необходимости)
- Добавить make-команды или скрипты:
  - `make run`
  - `make test`
  - `make lint`

## 12. CI-скелет
- Создать workflow (например, GitHub Actions):
  - установка зависимостей
  - lint
  - tests
- Без деплоя на первом этапе.

## 13. Документация для команды
- Создать `README.md` с разделами:
  - как поднять проект локально
  - структура проекта
  - как добавлять новую ручку (router -> schema -> service -> test)
  - правила именования и code-style
